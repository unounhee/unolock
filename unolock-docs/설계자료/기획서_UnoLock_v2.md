## 1. 서비스 개요

### 한 줄 정의

> **교재만 올리면, AI가 학생 맞춤 미션을 내고, 공부를 마친 만큼 스마트폰을 열어주는 학습 습관 형성 플랫폼.**

### 서비스의 본질

UnoLock은 기존 스마트폰 관리 앱의 일방적인 **"시간 제약"** 방식에서 벗어납니다. 단순히 폰을 빼앗는 것이 아니라, **학생이 학업을 마친 뒤 여가 시간을 갖는 "올바른 학습 습관"** 을 길러주는 것이 목적입니다.

핵심은 순서입니다. "공부 → 여가"라는 올바른 순서를 학생의 몸에 배게 만드는 것. 이것이 단순 차단 앱과 UnoLock이 근본적으로 갈리는 지점입니다.

### 누구를 위한 서비스인가

- **학원 / 과외 선생**: 학생 관리라는 부가 서비스를 자신의 상품에 연결
- **학부모**: 자녀가 어떤 진도를, 어떻게 학습하고 있는지 정보를 제공받음
- **학생**: 강제 통제가 아닌, 스스로 여가 시간을 버는 자기주도적 경험

## 2. 문제 정의 — 우리가 진짜 푸는 것

### 학부모의 고질적 문제

학부모가 겪는 가장 본질적인 고통은 **"우리 아이가 어떻게 공부하고 있는지 모른다"** 는 것입니다.

- 옆에서 봐도 무엇을 하는지 모른다
- 물어보면 아이는 신경질을 내고, 그러다 갈등이 생긴다
- 방치하다가 성적표를 보고 나서야 분노한다

이것은 단순한 정보 부족이 아니라, **정보 공백이 만들어내는 가정 내 갈등**입니다. UnoLock은 바로 이 지점을 메웁니다.

### 이것이 추측이 아닌 이유 — 이미 검증된 가치

이 사업의 핵심 근거는 **기획자(대표) 본인이 과외 시장에서 이미 검증했다**는 점입니다.

대표의 과외 상품은 **월 350 / 250 / 132만원** 3종으로 구성되어 있으며, 근 3년간 월 평균 약 1,200만원의 매출을 기록했습니다. 매우 높은 객단가임에도 학부모가 그만한 가치를 인정하고 결제하는 가장 큰 이유는 **"피드백 서비스"** 입니다.

> 교육 과정에 대한 전반적인 이해가 없는 학부모에게,
> **"학생의 수업 내용 → 문제점 → 해결 방안"** 을
> **"이해할 수 있는 수준"** 으로 풀어 제공한 것.

UnoLock의 본질은 여기서 나옵니다.

> **월 350만원짜리 1:1 피드백의 가치를, 소프트웨어로 복제해 객단가를 낮춘다.**

### 대표가 풀려는 자기 자신의 문제

과외 사업은 구조적 한계가 명확합니다.

- 객단가가 높을 수밖에 없어 **진입장벽이 큼**
- 학생 한 명이 들고 나는 것에 **매출 변동이 심함**
- 1:1 노동이라 **확장이 막혀 있음**

UnoLock은 이 노동을 **소프트웨어 레버리지**로 전환하는 시도입니다.

### 시장 검증 현황

- **학부모 반응**: 다수 학부모와 이미 논의 완료, 반응 긍정적
- **시범 학원**: 베타 테스트에 협조할 원장 다수 확보
- **유일한 변수**: 학원 설득 (단, 세일즈는 대표의 전문 영역)

## 3. 핵심 가치와 차별점

### 기존 경쟁 앱과의 결정적 차이

시장에는 이미 유사 컨셉의 앱들이 존재합니다 (StudyLock 등). 그러나 대부분 **"학생이 직접 자료를 올리고 혼자 푸는"** 구조입니다.

UnoLock의 차별점은 **학원(교사) 연동**입니다. 교사가 한 번 교재를 올리면 반 전체에 자동 배포되는 구조로, 결이 완전히 다릅니다.

### UnoLock을 지탱하는 4개의 기둥

| 기둥 | 내용 | 목적 |
|------|------|------|
| 1. 교사 노동 0 | 교사는 교재만 올린다, AI가 출제한다 | 도입 장벽 제거 |
| 2. 리추얼 | 미션은 시험이 아니라 매일의 습관 | 이탈 리스크 제거 |
| 3. 과목별 메커니즘 | 수학=재출제 루프, 영단어=떠먹이기 | 현장 적합성 |
| 4. 정보 비대칭 | 학부모는 성공만 본다 | 학원 이미지 방어 |

### 이 모든 것을 관통하는 한 문장

> **UnoLock은 학생을 평가하는 도구가 아니라, 학원의 신뢰를 지켜주는 도구다.**

## 4. 사용자 구조와 권한 설계

### 3대 권한 분리

UnoLock의 핵심 설계는 **권한을 3가지로 분리**하는 것입니다.

| 권한 | 보유 주체 | 내용 |
|------|-----------|------|
| **출제 권한** | 출제자 (학원 / 과외선생 / 직접학부모) | 교재 업로드, 미션 설정 |
| **시간관리 권한** | 학부모 | 보상 시간 설정, 사용 앱 범위 설정 |
| **열람 권한** | 출제자 ↔ 학부모 (범위 분리) | 학생 정보 조회 (범위가 다름) |

### 핵심: 열람 권한의 범위 분리

같은 학생 데이터라도 출제자(교사)와 학부모가 **보는 범위가 코드 레벨에서 갈립니다.** 이것이 뒤에 나올 "정보 비대칭"을 구현하는 기술적 토대입니다.

## 5. 미션 시스템 — 제품의 심장

### 미션의 철학: 시험이 아니라 리추얼

이 부분이 UnoLock 전체를 관통하는 가장 중요한 통찰입니다.

대부분의 교육 앱은 "더 어렵게, 더 빡세게"를 자랑으로 내세웁니다. 그러나 현장은 정반대입니다.

> **대부분의 학생은 공부를 열심히 하지 않는다.**
> 미션 난이도가 너무 높으면 → 학생이 폰을 못 풀고 → 학부모가 항의하고 → **학원이 욕을 먹는다.**

학원업의 핵심 메커니즘은 **"공부를 못하는 대부분의 학생을 얼마나 많이 데리고 가느냐"** 입니다. 따라서 미션은 학생을 거르는 장치가 아니라, **매일 책상 앞에 앉게 만드는 가벼운 트리거**여야 합니다.

> **UnoLock의 미션은 '시험'이 아니라 '리추얼(ritual)'이다.**

### 통과 기준의 이중성: "8할로 높되, 결국 통과시킨다"

언뜻 모순처럼 보이지만, 두 가지가 동시에 충족되어야 합니다.

| 요소 | 값 | 이유 |
|------|-----|------|
| **커트라인** | 8할(80%) 이상 | 학부모용 명분 ("기준이 빡셉니다") |
| **최종 결과** | 결국 통과 | 학원 리스크 방어 ("폰 못 풀어서 학원 탓 듣는 일 없게") |

미션의 진짜 목적은 **"못 푼 아이를 거르는 것"이 아니라, "떠먹여서라도 8할에 도달시키는 학습 루프"** 입니다. 통과 자체보다, 통과까지 가는 과정에서 한 번이라도 더 보고 풀게 만드는 것이 핵심입니다.

이 설계의 효과는 **학원의 고객 이탈 리스크를 0에 수렴**시킨다는 것입니다. "애 폰 못 풀게 만드는 앱"이 아니라 "애가 결국 통과하면서 공부도 되는 앱"이기 때문에, 학원이 안심하고 도입할 수 있습니다.

## 6. 과목별 메커니즘

미션은 과목에 따라 **완전히 다른 구조**로 세분화되어 배포됩니다.

### 수학 — "통과시키는 루프"

```
문항 풀이 → 8할 미달 → 해설 제공 →
같은 메커니즘의 다른 문항(다른 숫자/유형) 재출제 →
반복 → 결국 8할 통과
```

- 구성 예시: 개념 피드백(공식 정리) + 객관식 3문항 + 주관식 2문항
- 핵심: AI가 **"같은 개념, 다른 문항"** 을 계속 생성. 사람 교사는 못 하는, AI라서 가능한 영역.

### 영단어 — "떠먹이는 루프"

```
단어 10개 시험 → 틀린 단어 뜻 노출 →
10초 후 재시험 → 통과 유도
```

- 구성 예시: 수준별 지문 1개 + 영단어 10개
- 핵심: 아이가 맞춰서 통과할 수 있도록 적극적으로 유도

### 교사 조절 옵션 (반별 / 학원별)

같은 학원 안에서도 **반에 따라 다르게** 세팅 가능해야 합니다.

| 과목 | 조절 가능 항목 |
|------|----------------|
| 수학 | 문항 난이도 (상 / 중 / 하) |
| 영어 | 지문 포함 여부 + 지문 난이도 |

## 7. 정보 비대칭 설계

### 원칙: 정보의 비대칭을 의도적으로 설계한다

| 보는 사람 | 보이는 것 | 안 보이는 것 |
|-----------|-----------|--------------|
| **교사** | 모든 것 (틀린 횟수, 재시험 횟수, 취약 영역) | — |
| **학부모** | 통과 결과 + 깔끔한 리포트 | 몇 번 틀렸는지, 재시험 횟수 |
| **학생** | 자기 미션, 보상 | — |

### 학부모에게 전달되는 방식

학부모에게는 **최종적으로 통과한 내용만** 전달합니다.

> "지안 학생이 테스트를 통과했어요! 테스트 결과를 보여드릴까요?"

그 뒤의 시행착오(틀린 횟수, 재시험 횟수)는 가려줍니다.

### 왜 이 설계가 학원을 설득하는가

학원이 두려워하는 것은 **"우리 애가 못 따라간다는 사실이 학부모에게 노출되는 것"** 입니다. UnoLock은 그 노출을 구조적으로 막아주면서, 동시에 교사에게는 진짜 데이터(누가 취약한지)를 줘서 수업에 활용하게 합니다.

> **학원의 약점은 가려주고, 강점은 살려주는 도구.**

이 구조 덕분에 학원뿐 아니라 **과외 선생에게도 판매 가능**합니다.

## 8. 보상 구조 (스마트폰 잠금 흐름)

### 보상의 형태

스마트폰 잠금이 해제될 때, **UnoLock 앱이 화면을 잡는 구조**입니다. (잠금 해제 시마다 광고를 보여주고 보상을 주는 앱과 유사한 메커니즘.) 미션이 풀리지 않는다는 것은 곧 **앱을 빠져나갈 수 없는 상태**를 의미합니다.

### 전체 흐름

```
숙제 인증사진 올리기 (학부모가 볼 수 있어야 함)
        ↓
테스트 통과할 때까지 진행
        ↓
통과 → 잠금 해제
        ↓
학부모께 간단한 알림 전송
        ↓
매주 피드백 작성 후 보고 (학원이 설정·검토 가능한 형식)
```

### 미션 전(공부 전) 상태의 폰 제어

미션 전 상태에서 폰을 어떻게 제어할지는 **보호자가 선택**합니다.

- 보통은 **선택된 앱만 사용 가능**하도록 설정 (게임·유해 앱 설치 차단 포함)
- 옵션 A / B / C / 자유 형태로 제공 예정 (세부안은 추후 논의)

## 9. 수익 모델 — 두 개의 트랙

수익 구조는 **결제 주체에 따라 두 트랙**으로 명확히 갈립니다.

### 트랙 A — B2B (학원 / 과외 선생)

| 항목 | 내용 |
|------|------|
| 결제 주체 | 학원 또는 과외 선생 |
| 대상 학생 | 저학년 포함 전 학년 |
| 핵심 가치 | 본인 상품에 "관리 서비스"를 얹는 부가가치 |
| 핵심 기능 | AI 자동 출제 + 교사용 상세 데이터 |

### 트랙 B — B2C (직접학부모)

| 항목 | 내용 |
|------|------|
| 결제 주체 | 학부모 (직접 사용) |
| 마케팅 집중 타겟 | 저학년 (사람이 몰릴 것으로 예상되는 포인트) |
| 핵심 가치 | 폰 제약 + 기본 학습 관리 |
| 핵심 기능 | 학부모가 교재 보고 직접 퀴즈 등록 가능 |

### 두 트랙을 가르는 진짜 기준

결제 주체를 나누는 기준은 **"학부모의 교육 개입 가능 여부"** 입니다.

> "보통 중3 내용부터 학부모들이 손을 놓는다."

- 학부모가 직접 가르치고 검수할 수 있는 수준(저학년) → **B2C**
- 학부모가 손 못 대는 수준(중3+) → 전문가(학원)가 개입하는 **B2B**

단, 시스템상으로 학년을 제한하지는 않습니다. 저학년 직접학부모 시장은 **마케팅 포인트**로 별도 공략합니다.

### 같은 앱, 다른 모드

두 트랙은 사실 같은 앱의 **모드 차이**로 구현됩니다.

- **B2C 모드**: 학부모가 교재 올리고 퀴즈 등록 (출제자 = 학부모)
- **B2B 모드**: 교사가 교재 올리고 AI 출제 (출제자 = AI / 교사)

둘 다 "교재 → 미션 → 보상" 구조는 동일하며, **누가 출제 권한을 갖느냐**만 다릅니다.

## 10. 기술 구현 방향

### 플랫폼 우선순위: 안드로이드 먼저

스마트폰 제어 기능의 구현 가능 수준은 OS마다 다릅니다.

| 트랙 | 플랫폼 | 구현 방식 | 강제력 |
|------|--------|-----------|--------|
| B2C (직접학부모) | 안드로이드 | 기본 권한 + 앱 차단 | 강 |
| B2C (직접학부모) | iOS | Screen Time API | 중 (우회 가능성 존재) |
| B2B (학원 / 과외) | 양쪽 | MDM 키오스크 모드 | 매우 강 |

### 왜 안드로이드 먼저인가

- **안드로이드**: 앱 차단, 시간 제한, 앱 설치 승인, 승인 앱만 허용 등 비전을 100% 구현 가능
- **iOS**: 구현은 되나, 사용자가 설정에서 권한 토글을 끄면 제약이 풀리는 우회 구멍이 있음 (현재까지 미해결). 또한 잠금 화면에서 앱으로 복귀하는 흐름이 매끄럽지 않음
- 따라서 **안드로이드로 비전을 100% 구현한 뒤, iOS는 2차로 신뢰 기반 버전 제공**

### 예상 기술 스택 (초안)

| 구성 요소 | 방향 |
|-----------|------|
| 학생 / 학부모 앱 | 안드로이드 네이티브 우선 (시스템 권한 활용) |
| 출제자 대시보드 | 웹 기반 |
| AI 출제 엔진 | Claude API (교재 이미지/PDF → 문제 생성) |
| 백엔드 / DB | 서버 + 데이터베이스 (실시간 동기화) |
| 폰 제어 | 안드로이드 시스템 권한 / lock task mode |

## 11. 개발 전략 — "설계도 먼저" 철학

### 핵심 원칙

기능을 나중에 "붙이는" 것은 가능하지만, **아무렇게나 붙일 수는 없습니다.** 종이접기가 아니라 레고에 가깝습니다. 처음 설계도를 어떻게 그리느냐에 따라 나중에 붙이기가 쉬울 수도, 다 부수고 다시 만들 수도 있습니다.

### 전체 설계도를 먼저 그린다

```
[전체 설계도를 먼저 그림]
   ├─ 사용자 권한 구조 (3종 미리 자리 잡아둠)
   ├─ 미션 시스템 (통과 판정 자리 = 나중에 잠금이 연결될 지점)
   ├─ AI 출제 모듈
   └─ 폰 잠금 모듈 (자리만 비워둠)
         ↓
[1차: 잠금 빼고 나머지부터 완성]
         ↓
[2차: 비워둔 자리에 잠금 '끼워넣기']  ← 깔끔하게 연결됨
```

이렇게 설계도를 먼저 그려 자리를 비워두면, 나중에 큰 기능도 깔끔하게 끼워집니다. 설계도 없이 즉흥으로 만들면 나중에 무너집니다.

### 역할 분담 (바이브코딩)

- **대표(기획자)**: 기획과 의사결정에 집중. 코드를 몰라도 됨.
- **AI(개발 파트너)**: "설계도 관리자" 역할. 나중에 안 무너지게 짓는 책임.

## 12. 로드맵

### MVP 쪼개기 전략

처음부터 폰 잠금까지 넣지 않고, **검증 가능한 최소 단위**부터 만듭니다.

### 1차 MVP (약 4~6주) — "잠금 없는 핵심 루프"

```
교재 업로드 → AI 출제 → 학생 풀기 → 통과 → 학부모 알림
```

- 폰 잠금 없이 핵심 학습 루프만 완성
- **대표의 과외 학생들에게 즉시 테스트 가능**
- 학원 설득 시 "작동하는 것"을 보여주며 진행 가능

### 전체 베타까지 (약 3~4개월, 버퍼 포함)

| 단계 | 내용 | 기간 |
|------|------|------|
| 0. 학습 + 환경 세팅 | 개발 환경, 도구 설치, 기초 개념 | 1~2주 |
| 1. 핵심 루프 MVP | 교재 → AI 출제 → 풀기 → 통과 | 3~5주 |
| 2. 폰 잠금 기능 | 안드로이드 앱 차단 / 키오스크 | 2~4주 |
| 3. 권한 + 리포트 | 3종 권한 분리, 주간 리포트 | 3~4주 |
| 4. 다듬기 + 테스트 | 버그 수정, 실기기 테스트 | 2~3주 |

### 주의: 바이브코딩의 80/20 법칙

기능의 80%는 빠르게 만들어지지만, 나머지 20%(특히 **폰 잠금, 결제, 실기기 버그**)에서 시간의 80%가 소요됩니다. 안드로이드 시스템 권한은 에뮬레이터에서 되다가 실제 기기에서 안 되는 경우가 흔하므로, "폰 잠금"과 "테스트" 단계에 버퍼를 둡니다.

## 13. 부록 — 용어 정의

| 용어 | 정의 |
|------|------|
| **직접학부모** | 학원/과외를 거치지 않고, 학부모가 직접 출제자로서 UnoLock을 사용하는 경우 |
| **출제자** | 미션을 출제하는 주체 (학원 / 과외선생 / 직접학부모) |
| **리추얼(ritual)** | 학생을 거르는 시험이 아니라, 매일 책상 앞에 앉게 만드는 가벼운 학습 습관 단위 |
| **재출제 루프** | 미달 시 같은 개념의 다른 문항을 생성해 결국 통과시키는 수학 미션 구조 |
| **정보 비대칭** | 교사는 전체 데이터를, 학부모는 성공 결과만 보도록 의도적으로 분리한 설계 |

*앱 명칭 'UnoLock'은 가칭이며 언제든 변경 가능합니다.*
