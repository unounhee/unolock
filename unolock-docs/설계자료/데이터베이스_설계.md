# UnoLock 데이터베이스 설계 (표/테이블 도면)

> 확정일: 2026-06-23. 설계 근거: `기획서_UnoLock_v2.md`, `account_structure.svg`.
> 표는 "엑셀 시트 한 장"이라고 생각하면 됨. 아래는 어떤 시트를 어떤 칸으로 만들지 정한 도면.

## 전체 구조 — 3층 11개 시트

### 🟦 1층 — 사람과 공간 (계정·권한 기초)
| 시트 | 담는 것 | 핵심 칸 |
|---|---|---|
| `profiles` | 모든 사용자 1명당 1줄 + 역할 | role(teacher/student/parent) |
| `academies` | 학원·과외 "공간" | owner_id(만든 출제자) |
| `classes` | 반 (학원에 속함) | academy_id, name (난이도는 반 이름에 포함) |
| `memberships` | 학생 ↔ 반 연결 + 승인 | student_id, class_id, status |
| `guardianships` | 학부모 ↔ 학생 연결 (자녀증명) | parent_id, student_id, verify_code, status |

### 🟩 2층 — 콘텐츠 (교재·미션)
| 시트 | 담는 것 |
|---|---|
| `materials` | 업로드한 교재(사진/PDF 저장 위치) |
| `missions` | AI 미션 1개 (과목·난이도·통과기준 80%) |

### 🟨 3층 — 기록 (풀이·통과·알림) ← 정보 비대칭이 구현되는 곳
| 시트 | 담는 것 | 누가 봄 |
|---|---|---|
| `attempts` | 한 회차(시도): 회차번호·점수·통과여부·인증사진칸 | 교사 전체 / 학부모는 성공만 |
| `questions` | 회차별 문항 (정답 포함) — **각 회차(attempt)에 붙음** | 교사만 |
| `answers` | 문항별 학생 답 + 정오 | 교사만 |
| `notifications` | 학부모 통과 알림 | 학부모 |

## 핵심 연결 관계
- `학원(academies) → 반(classes) → 학생(memberships)`
- `학부모 → 학생` (guardianships)
- **재시험 루프**: `미션(missions) → 회차(attempts) → 문항(questions) → 답(answers)`
  - 회차마다 새 문항이 생성됨(수학) / 같은 문항 재시험(영단어)
  - `attempts.attempt_no` = 몇 번째 재시험인지 = 교사만 보는 정보 비대칭 데이터

## 정보 비대칭의 기술적 토대
- 학부모: `attempts`의 "통과✓" 결과 + `notifications`만 봄
- 교사: `answers`·`attempts`의 회차/틀린 과정 전부 봄
- → 보안 규칙(RLS)으로 시트별 "누가 무엇을 보는지"를 코드 레벨에서 분리

## 확정 시 내린 결정들
- 과목(수학/영단어)은 별도 시트 X → `missions`의 한 칸(subject)
- 직접학부모 = profiles(parent) + academies 소유 + guardianships로 처리 (별도 구조 불필요)
- 반(classes) 시트 **포함** (학원→반→학생)
- 숙제 인증사진 = `attempts`에 칸만 만들어둠(photo_url), 업로드 기능은 2차
- 폰잠금·보상지갑·구독결제 = 2차, 자리만 비워둠

## 만드는 순서
1층(이번) → 확인 → 2층 → 3층. SQL은 `supabase/migrations/`에 보관.
